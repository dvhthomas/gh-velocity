# Fix First-Run Experience and Strategy Completeness

## Overview

The most common new-user path — `preflight --write` on a repo without a project board — produces a config where cycle time is silently useless across every command. The default `strategy: issue` without `lifecycle.in-progress.project_status` guarantees N/A, with no warning in command output. This plan fixes the first-run experience by improving preflight heuristics, adding runtime guidance, cleaning up dead config, and updating documentation.

(see brainstorm: `docs/brainstorms/2026-03-12-strategy-completeness-audit-brainstorm.md`)

## Problem Statement / Motivation

1. **Silent N/A trap**: A user who runs `preflight --write` (no project board) gets `strategy: issue` with no lifecycle.in-progress. Every cycle-time call returns N/A with no explanation. This is the most common scenario.
2. **Inconsistent empty-result UX**: Lead-time and cycle-time bulk show "Verify:" URLs when empty, but report silently omits sections, and my-week shows no verify URL.
3. **Dead config**: `lifecycle.done.query` is generated by preflight and defaultConfigTemplate but never consumed by any command. Users who customize it see no effect.
4. **Outdated documentation**: docs/guide.md references old config field names (`project.id`, `statuses.backlog`, etc.) and the troubleshooting section doesn't cover the most common N/A cause.
5. **Missing "docs" category**: Preflight detects docs labels but excludes them from generated config.
6. **Report defect rate hardcodes "bug"**: Users who name their bug category differently get 0% defect rate with no explanation.

## Proposed Solution

### Phase 1: Fix Preflight Strategy Heuristic

Change the strategy inference in `cmd/preflight.go:295-303`:

```go
// Current (too conservative):
if !result.HasProject && result.RecentPRs > result.RecentIssues {
    result.Strategy = "pr"
}

// Proposed (prefer actionable data):
if result.HasProject {
    result.Strategy = "issue"
    // hint: project board detected, lifecycle config will enable cycle time
} else if result.RecentPRs > 0 {
    result.Strategy = "pr"
    // hint: no project board, using PR strategy for cycle time
} else {
    result.Strategy = "issue"
    // hint: no project board and no PRs — cycle time will be unavailable
}
```

**Rationale**: Issue strategy without a project board is guaranteed N/A. PR strategy produces useful data when any PRs exist. Only fall back to issue strategy (with prominent warning) when there are truly no PRs.

Update hint text for each branch to explain the reasoning and what the user can do to improve results.

### Phase 2: Add Runtime Warnings for Silent N/A

#### 2a: Cycle-time bulk pipeline warning

In `internal/pipeline/cycletime/cycletime.go`, `BulkPipeline.ProcessData()`:

After computing all items, if ALL items have nil duration, emit a strategy-specific warning:

```go
if len(durations) == 0 && len(p.Items) > 0 {
    switch p.StrategyStr {
    case "issue":
        p.Warnings = append(p.Warnings, "Cycle time unavailable for all issues — "+
            "no lifecycle.in-progress.project_status configured. "+
            "Add a project board: gh velocity config preflight --project-url <url>")
    case "pr":
        p.Warnings = append(p.Warnings, "Cycle time unavailable — "+
            "no issues had a closing PR. Ensure PRs reference issues with 'closes #N'.")
    }
}
```

This fires for BOTH strategies, since the user experience (all N/A) is the same regardless of root cause.

#### 2b: Single-issue cycle-time hint

In `internal/pipeline/cycletime/cycletime.go`, `IssuePipeline.ProcessData()`:

When the computed metric has nil Duration and nil Start (true N/A, not in-progress), add a warning:

```go
if p.CycleTime.Start == nil && p.CycleTime.Duration == nil {
    if p.StrategyStr == "issue" {
        p.Warnings = append(p.Warnings, "No cycle time signal — "+
            "configure lifecycle.in-progress.project_status for issue cycle time")
    }
}
```

#### 2c: Report and my-week N/A sections

In `internal/format/report.go`, when cycle time stats are nil, show "not available" with context instead of silently omitting the section:

```
Lead Time:    median 3.2d, mean 5.1d (n=14)
Cycle Time:   not available (no lifecycle signal configured)
Throughput:   15 issues closed, 8 PRs merged
```

In `internal/format/myweek.go`, when insights.CycleTime is nil but there are closed issues, add an insight line:

```
Cycle time not available — configure lifecycle.in-progress for cycle time metrics.
```

Report and my-week changes share the same pattern: show the section with an explanatory "not available" message rather than omitting silently. The `model.StatsResult` already carries `CycleTimeStrategy` and `Warnings` — pipe these through to the format layer.

### Phase 3: Clean Up Dead lifecycle.done.query

**Decision**: Stop generating `lifecycle.done.query` in new configs since it is never consumed by any command. Keep the struct field and parsing for backward compatibility.

Files to change:
- `cmd/preflight.go`: `renderPreflightConfig` — remove `lifecycle.done.query: "is:closed"` from non-project output path (lines 851-870)
- `cmd/config.go`: `defaultConfigTemplate` — remove lifecycle.done.query from template; add comment explaining that lifecycle stages use `project_status` for cycle time detection

**What stays unchanged**:
- `internal/config/config.go`: Keep `LifecycleStage.Query` field and parsing — existing user configs must still load without error
- Keep the default `Done.Query: "is:closed"` in `defaults()` — it's harmless and removing it could break `config show` output

**Backward compatibility**: Users with existing `lifecycle.done.query` in config get no behavior change (it was already ignored). No deprecation warning needed — just stop generating it in new configs.

### Phase 4: Improve Empty Results Guidance

#### 4a: My-week search URLs

Add search URLs to my-week pretty/markdown output for the lookback sections (closed issues, merged PRs). These are the two most actionable queries — my-week already has the `scope.Query` objects available. Surface them as "Verify:" links in the same pattern as lead-time and cycle-time bulk commands.

The 8 parallel queries each have different search URLs. Only surface the 2 lookback queries (issues closed, PRs merged) since those are what users most commonly want to verify. The lookahead queries (open issues, open PRs) are less surprising when empty.

### Phase 5: Preflight Minor Fixes

#### 5a: Add "docs" to category generation

In `cmd/preflight.go`, add `"docs"` to the category iteration loop (currently only `["bug", "feature", "chore"]`).

#### 5b: Document defect-rate "bug" convention

In docs/guide.md, add a note under the report/quality section explaining that defect rate counts issues classified as "bug". Users who name their category differently should use "bug" as the name.

### Phase 6: Update Documentation

#### 6a: Fix outdated config field names in docs/guide.md

Replace all references to old field names:
- `project.id` → `project.url`
- `project.status_field_id` → `project.status_field`
- `statuses.backlog` → `lifecycle.backlog.project_status`
- `statuses.active_labels` → (removed — explain lifecycle stages instead)
- `statuses.backlog_labels` → `lifecycle.backlog.project_status`

#### 6b: Add "Cycle time shows N/A for all issues" troubleshooting entry

Expand the troubleshooting section to cover:
- Issue strategy without lifecycle.in-progress → N/A (most common)
- No project board → suggest `preflight --project-url` or switch to PR strategy
- Project resolution failed → check URL, check token scopes

#### 6c: Add "Which strategy should I pick?" decision guide

Add a section to docs/guide.md:

```markdown
### Choosing a cycle time strategy

| Your workflow | Recommended strategy | Why |
|---------------|---------------------|-----|
| Issues tracked on a project board | `issue` | Measures real work time (board status change → closed) |
| PRs close issues (most OSS repos) | `pr` | Measures PR review time (created → merged) |
| Issues only, no project board or PRs | `issue` | Lead time works; cycle time requires project board |

If you're unsure, start with `pr` — it works immediately with no extra config.
To enable `issue` strategy cycle time, add a project board and run:
  gh velocity config preflight --project-url <your-project-url> --write
```

#### 6d: Add "No results?" troubleshooting entry

```markdown
### No results / empty output

1. **Check your date range**: `--since 30d` looks at the last 30 days. Try a wider range.
2. **Check your scope**: Run with `--debug` to see the GitHub search query and verify URL.
3. **Check the verify URL**: Bulk commands show a "Verify:" link — open it in GitHub to see what the search returns.
4. **Check for activity**: A repo with no closed issues or merged PRs in the window will show empty results. That's correct.
```

## Dependencies & Risks

| Risk | Mitigation |
|------|-----------|
| Changing default strategy for no-project repos is a behavior change | Only affects newly generated configs; existing configs keep their strategy |
| Users may expect lifecycle.done.query to work | It never worked; removing it from templates is a net improvement |
| Outdated guide sections may have links from external sources | Keep redirects/anchors stable when restructuring |
| Phase 2 warnings could be noisy | Warnings fire once per command invocation, not per item |
