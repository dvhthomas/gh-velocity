# Plan: Fix GitHub Search URLs â€” date format and verify URL accuracy

**Complexity:** Standard
**Risk:** Medium â€” changes query format used by both API data fetching and user-facing URLs

## Problem

GitHub Search URLs generated by `scope.Query.URL()` never return correct results when clicked. Two root causes:

### Bug 1: RFC3339 timestamps cause boundary misses

All date-qualified queries use `time.RFC3339` (e.g., `closed:2026-03-15T16:15:14Z..2026-03-22T16:15:14Z`).

**Evidence:**
- `merged:2026-03-15T00:00:00Z..2026-03-22T00:00:00Z` â†’ 54 results
- `merged:2026-03-15..2026-03-22` â†’ 59 results

Date-only format treats the end date inclusively ("through end of day"), while timestamps are exclusive at the exact second. PRs merged after the exact timestamp are missed. Non-midnight timestamps (e.g., `16:15:14Z`) exclude even more.

**Important:** The same `time.RFC3339` format is used in both:
- API queries (data fetching via `q.Build()`) â€” these *work* because they use the exact `since`/`until` boundaries and results are consistent with what was fetched
- Verify URLs (via `q.URL()`) â€” these *break* because clicking them re-runs the search with slightly different boundary semantics

### Bug 2: `reviewed-by:` qualifier returns 0 results

`reviewed-by:dvhthomas` returns 0 results via GitHub Search API, even without date filtering. This is a known limitation â€” `reviewed-by:` is not a supported GitHub Search qualifier (it doesn't appear in GitHub's search syntax docs). This affects both data fetching and URLs.

**This bug is pre-existing and out of scope for this PR.** The `reviewed-by:` query was already returning 0 results before this change. Fixing it requires switching to the GraphQL API to find review activity, which is a separate feature.

## Solution

### Change: Use date-only format for URL generation only

The fix is narrow: change `Query.URL()` to use date-only format (`YYYY-MM-DD`) instead of RFC3339 timestamps. This ensures verify URLs match what users see when they click through to GitHub Search.

**Do NOT change `Query.Build()`** â€” the API queries use RFC3339 intentionally to match the exact time boundaries used for data fetching.

### Implementation

Add a `URLQuery()` method to `Query` that reformats date qualifiers from RFC3339 to date-only format. The `URL()` method calls `URLQuery()` instead of `Build()`.

**Pattern:** Use a regex to find date range patterns like `closed:2026-03-15T16:15:14Z..2026-03-22T16:15:14Z` and replace with `closed:2026-03-15..2026-03-22`.

Specifically, match: `(closed|merged|updated|created):(\d{4}-\d{2}-\d{2})T[^\s.]+\.\.(\d{4}-\d{2}-\d{2})T[^\s]+`
Replace with: `$1:$2..$3`

This handles all date qualifiers uniformly without changing the Query struct or any callers.
