# RFC3339 Timestamps in GitHub Search Verify URLs

## Problem

Every verify URL generated by `scope.Query.URL()` returned incorrect (incomplete) results when clicked in a browser. The URLs looked correct but consistently showed fewer items than the CLI had actually fetched.

Example broken URL:
```
https://github.com/search?q=repo:dvhthomas%2Fgh-velocity%20is:pr%20is:merged%20merged:2026-03-15T16:15:14Z..2026-03-22T16:15:14Z
```

## Root Cause

GitHub Web Search handles date-only format and RFC3339 timestamps differently:

- **Date-only** (`merged:2026-03-15..2026-03-22`): End date is **inclusive through end of day**
- **RFC3339** (`merged:2026-03-15T00:00:00Z..2026-03-22T00:00:00Z`): End boundary is **exclusive at the exact second**

The `Query.URL()` method called `Query.Build()` which formatted dates as RFC3339 (`time.RFC3339`). This was correct for the REST API data-fetching path (where timestamps match the exact boundaries used for data collection), but wrong for user-facing verify URLs where inclusive end-of-day semantics are expected.

**Evidence**: Same query, two formats:
- `merged:2026-03-15T00:00:00Z..2026-03-22T00:00:00Z` → **54 results**
- `merged:2026-03-15..2026-03-22` → **59 results**

5 PRs merged on 2026-03-22 after midnight UTC were excluded by the timestamp format.

## Investigation Steps

1. Checked GitHub docs — confirmed ISO 8601 with Z suffix is supported syntax
2. Verified URL encoding was correct (`url.PathEscape`, not `QueryEscape`)
3. Tested actual API responses with both formats — found 54 vs 59 result discrepancy
4. Identified the semantic difference: date-only is end-inclusive, timestamps are end-exclusive
5. Confirmed `Build()` must keep RFC3339 for API precision — only `URL()` needs date-only

## Solution

Added a regex to `Query.URL()` that strips RFC3339 timestamps to date-only format. `Build()` was left unchanged so API queries keep their precise boundaries.

```go
// internal/scope/scope.go

var dateRangeRe = regexp.MustCompile(
    `(closed|merged|updated|created):(\d{4}-\d{2}-\d{2})T[^\s.]+\.\.(\d{4}-\d{2}-\d{2})T\S+`)

func (q Query) URL() string {
    query := q.Build()
    if query == "" {
        return ""
    }
    query = dateRangeRe.ReplaceAllString(query, "${1}:${2}..${3}")
    return "https://github.com/search?q=" + url.PathEscape(query)
}
```

The regex matches any date-range qualifier (`closed:`, `merged:`, `updated:`, `created:`) followed by RFC3339 timestamps separated by `..`, and replaces with date-only format preserving the qualifier name and date portions.

| Before | `merged:2026-03-15T16:15:14Z..2026-03-22T16:15:14Z` | 54 (incomplete) |
| After | `merged:2026-03-15..2026-03-22` | 59 (correct) |

## Prevention

### Key Principle: API Precision vs Presentation Are Different Concerns

Keep internal representations in API-native format (RFC3339). Convert to user-facing format only at output boundaries (URLs, logging, UI). Never assume that the format correct for an API call is also correct for a user-facing URL — test both paths.

### When Adding New Date-Qualified Queries

1. Use `Query.Build()` for API calls (RFC3339 precision)
2. Use `Query.URL()` for user-facing links (date-only, inclusive)
3. Add the new qualifier name to the `dateRangeRe` regex alternation if needed
4. Write a test case in `TestQuery_URL_DateOnly` covering the new query type
